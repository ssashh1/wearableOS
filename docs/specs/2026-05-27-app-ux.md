# OpenWhoop App UX — Spec & Findings (2026-05-27)

Execution record for `docs/plans/2026-05-27-app-ux-plan.md`: a WHOOP-like consumer iOS app
on top of the working pipeline + server-computed metrics. Built subagent-driven with two-stage
review, verified in the iOS Simulator (iPhone 17) against the **live** server, and end-to-end
checked with Maestro flows.

## 1. Tab / screen map

| Tab | Contents |
|---|---|
| **Today** | Recovery ring (% + band color, "Pending" on cold-start), Day Strain (0–21), Last-night sleep one-liner, HRV, Resting HR — all from the server cache. Live HR + strap battery appear when the strap is connected. Clean nav bar (no buttons). Each metric card taps → its MetricDetailView. |
| **Sleep** | Headline (efficiency % + time asleep + bed→wake), **hypnogram** (Awake/REM/Light/Deep bands), stage breakdown (deep/rem/light/in-bed/disturbances/latency), in-sleep signals (RHR/HRV/Resp/SpO2/Skin-temp), **7-night fall-asleep/wake** chart (real clock times per night), and a **Smart Alarm** card → AlarmView. |
| **Trends** | Unified line/bar charts for Recovery / HRV / RHR / Strain / Sleep duration over 7/30/90 d; each card → MetricDetailView. A **Heart Rate** card (raw 1 Hz, last 24 h) → HeartRateDetailView. |
| **Workouts** | Auto-detected workouts list (duration, avg HR, strain, calories) → WorkoutDetailView (HR-zone breakdown, peak HR, strain, calories, %HRR). |
| **Device** | Technical console (connection/bond/battery, live HR, command buttons, haptics, battery alerts, raw toggle, log) + a **Body Profile & Settings** card → SettingsView + a Test-Alarm buzz. |

**Entry points:** Settings lives on **Device** (config home); Alarm lives on **Sleep** (wake belongs with sleep). Neither clutters Today's nav bar.

## 2. Architecture (one data contract, two renderers)

- **Server is the single source of truth.** The phone does NO metric computation. `MetricsCache.swift` (formerly misnamed `DerivedMetrics.swift`; **kept, renamed**) is the offline GRDB cache of server metrics. `MetricsRepository` (a `@MainActor` facade, injected app-wide) reads the cache and triggers `ServerSync.pullDerived()` (derived-only, fast) — works in the sim over the network with no strap.
- **Unified chart system:** `Charts/MetricChart.swift` (one reusable Swift Charts component: line/bar, recovery banding, iOS-16 tap-selection via `.chartOverlay`+`proxy.value(atX:)`, clipped to card bounds, adaptive x-axis labels) is used by both the in-card previews and `Charts/MetricDetailView.swift` (full-page: range selector 1W/1M/3M/6M, AVG/MIN/MAX/LATEST stats strip, chevron paging). `HeartRateDetailView` is the stream-backed variant (6H/24H/3D/7D).
- **Single shared `BLEManager`** (owned by `AppRoot` via `LiveViewModel`, injected) used by both the Device tab and the alarm — exactly one `CBCentralManager`/state-restoration session, created at launch.
- **WhoopStore** is GRDB with **WAL + busy-timeout** so the Device-tab handle and the repository handle coexist safely on-device.

## 3. Shared server API contract (`https://whoop.example.com`, Bearer auth, device `my-whoop`)

| Endpoint | Use |
|---|---|
| `GET /v1/daily?device&from&to` | Daily metrics (recovery/strain/sleep mins/HRV/RHR/spo2/skin-temp/resp/sleep_start/end). Cached. |
| `GET /v1/sleep?device&date` | Sleep session(s) for the night ending `date`: stages (hypnogram), efficiency, RHR, HRV. Cached. |
| `GET /v1/streams/hr?device&from&to&max_points` | Downsampled raw HR for the Heart-Rate views. |
| `GET /v1/workouts?device&from&to` | Auto-detected workouts (see §5). |
| `GET/POST /v1/profile` | Body profile (height_cm, weight_kg, age, sex) for calorie/strain analysis. |
| `POST /v1/backfill-workouts {device,from,to}` | Recompute workouts (with calories) over a range — triggered after a profile save. |
| `POST /v1/compute-daily`, `GET /v1/summary`, `GET /v1/streams/{kind}` | Existing. |

## 4. Smart alarm — design + APK findings (reverse-engineered)

Command codes (confirmed for interoperability): `SET_ALARM_TIME=66`, `GET_ALARM_TIME=67`,
`RUN_ALARM=68`, `DISABLE_ALARM=69`, `RUN_HAPTICS_PATTERN=79`.

Payloads (Rev1, little-endian; implemented + unit-tested in `AlarmCommandsTests`):
- **SET_ALARM_TIME(66):** `[0x01] + <u32 epoch_s LE> + <u16 subseconds=0>` (7 bytes). Arms the strap's **firmware** alarm (event `STRAP_DRIVEN_ALARM_EXECUTED=57`) — fires even if the app is backgrounded/killed. RTC must be UTC-correct → we `SET_CLOCK` first.
- **GET_ALARM_TIME(67):** `[0x01]`. **RUN_ALARM(68):** `[0x01]`. **DISABLE_ALARM(69):** `[0x01]`.
- App-driven buzz uses `RUN_HAPTICS_PATTERN(79)` preset id=2 (the official app's alarm pattern; verified to buzz on-device). A 12-byte explicit-waveform form also exists in the APK (`[wfe1=47,wfe2=152,…,overall_loop=7,dur=30]`) — noted as an on-device alternative.
- **Smart wake** (best-effort): within the window, `ENTER_HIGH_FREQ_SYNC(96)` `[0x02]+<u16 interval LE>+<u16 duration LE>` (e.g. 180 s / 7200 s = `[0x02,0xB4,0x00,0x20,0x1C]`), watch HR/movement for near-wake, fire the alarm, then **always `EXIT_HIGH_FREQ_SYNC(97)`** (prior research flagged high-freq-sync as a data-loss risk if parked). WHOOP's wake decision is server-side; OpenWhoop uses a local HR heuristic + the firmware alarm as the guaranteed fallback.

**Design:** fixed-time firmware alarm is the guaranteed path (shipped + byte-verified); smart-wake is a conservative, window-bounded, EXIT-safe scaffold. **Needs on-device testing:** actual buzz, firmware-alarm persistence across disconnect/kill, smart-wake background-BLE behavior. (Cannot be verified in the simulator.)

## 5. Workout auto-detection method

Detection ALREADY existed server-side (`analysis/exercise.py::detect_exercises`): bouts from 1 Hz HR
(floor = RHR+15 bpm) gated by gravity-derived motion (>0.20 g/s), ≥5 min, 60 s gap-merge →
per-bout avg/peak HR, Edwards HR-zone time %, strain (0–21), HRmax. **Added:** calorie estimation
(`analysis/calories.py`) using the published **Keytel et al. 2005** energy-expenditure equation
(per-second active rate `(hr·c1 + kg·c2 + age·c3 + α)/251.04`, resting Harris–Benedict below 30 % HRR; needs the
profile), the `/v1/workouts` read endpoint, and `/v1/backfill-workouts`. Calories are `None` until a
profile is set (additive — no change to validated metrics otherwise).

## 6. Deploy

Use the host manifest (rootless Docker, `DATA_ROOT=/home/jp/data`):
`ssh jpserver 'cd ~/home-server && set -a; . hosts/$(hostname -s).env; set +a; docker compose -f stacks/whoop/docker-compose.yml up -d --build'` (or `make up`). **Do not** run bare `docker compose` without sourcing the manifest — `DATA_ROOT` resolves empty → `/whoop/db` → rootless mkdir fails → the `--build` recreate takes the stack down.

## 7. Known limitations / on-device TODO

- Recovery cold-starts (~4 nights) → "Pending" until enough history.
- SpO2/skin-temp/resp are WHOOP-*like* approximations (uncalibrated).
- Calories require the user's body profile (Settings) + a backfill.
- BLE-dependent features (live HR/battery on Today, the Device console connect, the alarm buzz +
  firmware persistence, smart-wake) are code-complete + byte-verified but need a physical strap.
- Workout detection currently surfaces some short low-intensity bouts; list filtering/tuning is a
  future refinement.

## 8. Manual test steps (after installing on the phone)

1. **Today:** open the app → recovery ring + strain + last-night sleep + HRV/RHR show your real numbers; tap **HRV** → detail page → switch 1W/1M/3M/6M (chart updates). Back.
2. **Sleep:** open Sleep → hypnogram + headline (efficiency/duration) + stage breakdown + in-sleep signals; scroll → 7-night chart shows each night's real bed→wake times. Tap **Smart Alarm** → set a wake time → Set alarm (with the strap connected, confirm it buzzes at that time; smart-wake = best-effort).
3. **Trends:** open Trends → 7/30/90-day charts; tap any card → full history + range; scroll to **Heart Rate** → tap → 6H/24H/3D/7D raw HR.
4. **Workouts:** open Workouts → your detected workouts; tap one → HR-zone breakdown + strain. (After setting your profile, calories populate.)
5. **Device:** open Device → connect the strap (live HR + battery should appear on Today too); tap **Body Profile & Settings** → enter height/weight/age/sex → Save (calories backfill kicks off). Test the alarm buzz button.
